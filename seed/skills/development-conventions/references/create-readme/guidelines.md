## README Writing Guidelines

### Formatting Rules

1. **No emoji overuse** — keep the readme concise and to the point. One or two emojis maximum, only if they add value.
2. **No sections for LICENSE, CONTRIBUTING, CHANGELOG** — there are dedicated files for those. Don't duplicate.
3. **Standard Markdown** with admonition syntax (`> **Note:**`) where appropriate.
4. **Logo/icon** — if you find a logo or icon for the project, use it in the readme's header.
5. **Tone** — professional, informative, direct. Open-source friendly but not overly casual.
6. **Structure** — follow well-structured open source READMEs. Common sections: title + badge bar, description, quick start, usage, configuration, API (if library), contributing link, license link.
