# API Design Conventions

## When to Use

Invoke this skill when designing or implementing REST/HTTP APIs — routes, handlers, controllers, endpoints (`**/{routes,handlers,api,controllers,endpoints}/**/*`).

## HTTP Status Codes — Be Precise

Return the most specific status code available. Don't default to 200 or 500.

```text
2xx — Success
  200 OK            — Standard success (GET, PATCH)
  201 Created       — Resource created (POST). MUST include Location header
  202 Accepted      — Async processing started. Return status endpoint
  204 No Content    — Success, no body (DELETE)

3xx — Redirection
  301 Moved Permanently
  304 Not Modified  — Cached response still valid (ETag/If-None-Match)

4xx — Client Error
  400 Bad Request   — Malformed input (validation errors)
  401 Unauthorized  — Missing or invalid credentials
  403 Forbidden     — Authenticated but not authorized
  404 Not Found     — Resource doesn't exist
  409 Conflict      — Resource state conflict (duplicate, version mismatch)
  422 Unprocessable — Semantic validation failure (well-formed but wrong)
  429 Too Many Requests — Rate limit exceeded

5xx — Server Error
  500 Internal Error — Unexpected failure (bug). Never return by default
  502 Bad Gateway   — Upstream returned invalid response
  503 Unavailable   — Temporarily down (maintenance, overload)
  504 Gateway Timeout — Upstream didn't respond in time
```

- **Never return 500 by catching and swallowing.** 500 means "bug."
- **Never return 200 with an error message.** Breaks every HTTP client, cache, and monitoring system.
- **401 vs 403**: 401 = "who are you?" (missing auth). 403 = "I know who you are, but no."

## Error Response Shape — Standardize

Every error response MUST use the same structure.

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Human-readable description of what went wrong",
    "details": [
      {
        "field": "email",
        "reason": "must be a valid email address",
        "value": "not-an-email"
      }
    ],
    "requestId": "req_a1b2c3d4"
  }
}
```

- **`code`**: machine-readable, stable, uppercase with underscores
- **`message`**: human-readable, safe to show in UI
- **`details`**: array of field-level errors for validation failures
- **`requestId`**: correlation ID for debugging
- **Never leak internals**: No SQL errors, stack traces, file paths

## Versioning

```text
# URL path versioning (most common, simplest to cache/route)
GET /api/v1/users

# Header versioning (cleaner URLs, harder to test in browser)
GET /api/users
Accept: application/vnd.myapp.v2+json
```

- **URL path versioning** is the safe default
- **Major version only**: `v1`, `v2`, not `v2.1.3`
- **Never break a published version**
- Deprecation: set `Sunset` and `Deprecation` headers

## Authentication & Authorization

```text
Authorization: Bearer <token>
```

- Auth middleware runs before any handler logic
- Rate limiting runs before auth (prevent brute force)
- Auth tokens: short-lived access (15 min) + longer refresh (7 days)
- 401 for missing/invalid credentials, 403 for insufficient permissions
- Never differentiate 401/403 based on resource existence (prevents enumeration)

## Pagination — Mandatory

Every list endpoint MUST paginate.

```json
{
  "data": [...],
  "pagination": {
    "cursor": "eyJsYXN0X2lkIjogNDJ9",
    "hasMore": true,
    "total": 1543
  }
}
```

- **Cursor-based**: preferred for large datasets
- **Offset-based**: simpler for small-to-medium datasets
- **Default limit**: 20-50 items. Max limit: 100
- **Always return `hasMore` or `total`**

## Rate Limiting

```text
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 987
X-RateLimit-Reset: 1687132800
```

- Per-user and per-IP rate limiting
- Return 429 + `Retry-After` header

## Idempotency

`PUT` and `DELETE` must be idempotent. `POST` should use idempotency keys for payment/creation operations.

```text
POST /api/v1/charges
Idempotency-Key: 8f7b3c2a-9e4d-11ed-a1eb-0242ac120002
```

- Idempotency keys stored for at least 24 hours
- Keys are UUIDs generated by the client

## Request/Response Conventions

- **JSON only**: `Content-Type: application/json`
- **ISO 8601 timestamps**: `"2026-06-18T14:30:00Z"`. Always UTC.
- **Snake_case or camelCase**: pick one and enforce it everywhere
- **No envelope for single resources**, envelope for collections
- **Boolean fields**: `isActive`, `hasSubscription`
- **Empty arrays over null**: `"items": []` not `"items": null`

## HTTP Methods — Use Correctly

```text
GET     /users          — List users
GET     /users/42       — Get user 42
POST    /users          — Create a user
PUT     /users/42       — Replace user 42 (full replacement)
PATCH   /users/42       — Update user 42 (partial update)
DELETE  /users/42       — Delete user 42
```

- **Never use GET for mutations**
- **POST for creation**, return 201 + Location header
- **PUT for full replacement**, PATCH for partial updates
- **No verbs in URLs**: `POST /users` not `POST /create-user`

## Documentation

Every API needs an OpenAPI spec (`openapi.yaml` or `openapi.json`). Not optional.
- Document every endpoint, every status code, every field
- Examples for all request/response bodies
- Spec is source of truth, not the docs page
